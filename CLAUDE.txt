## 3. LangGraph Agent Pipeline

The agent is a **7-node linear StateGraph** compiled once per analysis run:

```
understand_data → clean_data → eda → feature_engineering → train_models → evaluate_models → explain_results → END
```

### AgentState (TypedDict)
| Field | Type | Purpose |
|-------|------|---------|
| `df` | `pd.DataFrame` | Working dataset (mutated by cleaning) |
| `problem_type` | `str` | `"Regression"` / `"Classification"` / `"Clustering"` |
| `target` | `Optional[str]` | Target column name (None for clustering) |
| `X`, `y` | DataFrame / Series | Features and target after FE |
| `X_train/X_test/y_train/y_test` | Optional splits | 80/20 split for supervised |
| `metrics` | `Dict[str, Any]` | Per-model evaluation scores |
| `model_name` | `str` | Best model name |
| `best_params` | `Dict` | Non-default hyperparameters per model |
| `eda_results` | `Dict` | Stats, correlations, plot plan metadata |
| `eda_figures` | `List[Figure]` | Rendered matplotlib figures |
| `explanation` | `str` | LLM-generated narrative |
| `cleaning_report` | `str` | Data cleaning log |
| `feature_report` | `str` | Feature engineering log |
| `models` | `Dict[str, Any]` | Fitted model objects |

### Node Details

1. **`understand_data`** — LLM infers problem type from data sample; heuristic fallback (>10 unique numeric values → Regression)
2. **`clean_data`** — Normalizes pandas 2.x extension dtypes, drops duplicates, fills missing (median/mode), LabelEncodes categoricals
3. **`eda`** — AI-guided: tries Anthropic → Groq → deterministic fallback for plot planning; renders up to 18 plots from 20 supported types
4. **`feature_engineering`** — One-hot encoding, interaction terms (top 2 correlated), polynomial features (top feature squared)
5. **`train_models`** — RandomizedSearchCV tuning; Clustering uses `_best_k_by_silhouette` for auto-k
6. **`evaluate_models`** — Best model selection (R² → Regression, F1 → Classification, Silhouette → Clustering); refits best on full data for supervised
7. **`explain_results`** — LLM generates stakeholder-friendly narrative

## 4. ML Model Suite

### Regression (14 models)
Linear Regression, Ridge, Lasso, ElasticNet, SVR, KNN Regressor, Decision Tree, Random Forest, Gradient Boosting, AdaBoost, XGBoost, LightGBM, CatBoost, Neural Network (MLP), Voting Ensemble

### Classification (13 models)
Logistic Regression, SVM, KNN Classifier, Naive Bayes, Decision Tree, Random Forest, Gradient Boosting, AdaBoost, XGBoost, LightGBM, CatBoost, Neural Network (MLP), Voting Ensemble

### Clustering (6 models)
KMeans (auto-k via silhouette), Agglomerative (4 linkages), DBSCAN (grid search eps/min_samples), MeanShift (bandwidth tuning), Birch, Gaussian Mixture (BIC selection)

### Training Conventions
- All models wrapped in `Pipeline(preprocessor → model)` with `StandardScaler` on numeric columns
- Hyperparameter tuning via `RandomizedSearchCV` with adaptive `n_iter` (capped at search space size)
- CV folds: `max(2, min(5, n_rows // 10))` for regression; `max(2, min(5, min_class_count))` for classification
- `DEFAULT_RANDOM_STATE = 42` used everywhere
- Ensemble (Voting) built from top 5 candidate models if ≥2 are available
- Optional imports: XGBoost, LightGBM, CatBoost handled via try/except; gracefully skipped if not installed

### Evaluation Metrics
| Problem Type | Primary Metric | All Metrics |
|--------------|---------------|-------------|
| Regression | R² | MSE, R², MAE |
| Classification | F1 (weighted) | Accuracy, Precision, Recall, F1 |
| Clustering | Silhouette Score | Clusters, Noise Points, Silhouette, Calinski-Harabasz, Davies-Bouldin |

## 5. API Contract

**Base path:** `/api` (all routes prefixed)

| Method | Endpoint | Auth | Rate Limit | Purpose |
|--------|----------|------|------------|---------|
| `POST` | `/upload` | No* | 10/min | Upload CSV, get session_id + DatasetInfo |
| `POST` | `/configure` | Firebase | — | Set target, problem_hint, test_size, random_seed, scaling |
| `POST` | `/analyze` | Firebase | 5/min | Start async pipeline (background task), returns immediately |
| `GET` | `/status/{session_id}` | Firebase | — | Poll task status: idle/running/completed/failed |
| `GET` | `/results/{session_id}` | Firebase | — | Fetch full analysis results after completion |
| `POST` | `/predict` | Firebase | — | Single-row prediction using best model |
| `POST` | `/chat` | Firebase | 30/min | Session-scoped AI chatbot with plot generation |
| `GET` | `/sessions` | Firebase | — | List user's session IDs |
| `DELETE` | `/sessions/{session_id}` | Firebase | — | Delete a session |
| `GET` | `/health` | No | — | Returns `{"status": "ok", "timestamp": ...}` |

*Upload is rate-limited but not auth-gated to allow the initial upload flow.

### Session Ownership
- Sessions are bound to Firebase `uid` on first authenticated access
- `_enforce_session_access()` prevents cross-user access (HTTP 403)
- In-memory storage via `SessionStore` (asyncio-locked dict) — **not persistent across restarts**

### Chat Sandbox
- LLM-generated plot code runs in an **isolated subprocess** (`subprocess.run` with 45s timeout)
- DataFrame is serialized as JSON (first 5000 rows), base64-encoded and injected
- Matplotlib Agg backend, output saved as temp PNG → base64 response
- Optional memory limit via `PLOT_SANDBOX_MEMORY_MB` env var

## 6. EDA Plot System

The EDA module supports **20 plot types**: histogram, kde, boxplot, violinplot, barplot, countplot, scatterplot, lineplot, heatmap, pairplot, piechart, stackedbar, missingvalues, outlierdetection, distribution_grid, target_vs_feature, class_balance, regression_residual, feature_importance_proxy, pca_scatter

### Plot Plan Source Priority
1. **Anthropic** (Claude) — if `ANTHROPIC_API_KEY` set
2. **Groq** (Llama 3.3 70B) — if `GROQ_API_KEY` set
3. **Deterministic fallback** — rule-based plan based on data profile

All AI-generated plans are validated against `SUPPORTED_PLOT_TYPES` and sanitized. If AI plan produces zero renderable figures, falls back to deterministic.

3. **🏆 Model Performance** — Bar chart comparison, metrics table, best model highlight

### State Flow
```
HomePage.onUpload → sessionId + datasetInfo
   ↓
ConfigurePage.onAnalyze → polls /status → fetches /results → AnalyzeResponse
   ↓

```

### Auth Flow
```

   ↓ setAuthTokenGetter
api.ts authHeaders() → Bearer token injection on every request
```

## 9. Key Design Decisions & Gotchas

### Backend
- **Async pipeline**: All pipeline nodes are `async def` wrappers that delegate CPU work to `asyncio.to_thread()`. The graph runs via `graph.ainvoke()`.
- **Session storage is in-memory**: Sessions (DataFrames, trained models) live in `SessionStore._sessions` dict. A Render restart loses all data. The codebase explicitly notes "Replace with Redis / DB for production."
- **Best model refitting**: After evaluation, the best supervised model is re-fitted on the **full dataset** (train+test) for optimal prediction quality. Evaluation scores still reflect the honest train/test split.
- **Pandas 2.x compatibility**: `data_cleaning.py` explicitly normalizes nullable extension types (StringDtype, BooleanDtype, Int64Dtype) to NumPy-compatible equivalents before processing.
- **LabelEncoder reuse in predict**: The `/predict` endpoint reconstructs the full pipeline by concatenating the new row with the original dataset, cleaning/FE-ing the combined data, then extracting the last row. This ensures feature alignment.

### Frontend
- **No state persistence**: All app state (sessionId, datasetInfo, result) lives in React `useState` — a page refresh resets. Navigation guards redirect to `/` if state is missing.
- **3D Spline**: The homepage embeds a Spline 3D robot scene (`@splinetool/react-spline`).
- **Auth token auto-injection**: `AuthContext` wires `getIdToken()` into `api.ts` via `setAuthTokenGetter()` on mount.

### LLM
- **Temperature**: `understand_data` uses 0, `explain_results` uses 0.7, `chat` uses 0.3, `eda` planning uses 0.1
- **Fallback heuristics**: If LLM fails or hallucinates, both `understand_data` and `eda` have deterministic fallback logic

## 10. Development Workflow

### Start Backend
```bash
cd backend
python -m venv .venv && source .venv/bin/activate
pip install -r requirements.txt
# Create .env with GROQ_API_KEY and Firebase SA
uvicorn main:app --reload
```

### Start Frontend
```bash
cd frontend
npm install
# Create .env with VITE_* Firebase config
npm run dev
```
Frontend runs on `http://localhost:5173`, Backend on `http://localhost:8000`.

### Deployment
- **Backend** → Render (Docker or native Python service via `render.yaml`)
- **Frontend** → Vercel (static SPA, `vercel.json` handles SPA rewrites)

## 11. Critical Patterns for Contributors

1. **Adding a new ML model**: Add to `_train_regression()` / `_train_classification()` / `_train_clustering()` in `pipeline/model_training.py`. Always wrap in `_make_pipeline()` for scaling. Use `_tune()` for hyperparameter search with try/except guard.

2. **Adding a new EDA plot type**: Add the type string to `SUPPORTED_PLOT_TYPES` set, add the rendering case in `_execute_plot()`, and optionally add to `_default_plot_plan()` for deterministic fallback.

3. **Adding a new API endpoint**: Add Pydantic models to `api/schemas.py`, add route handler to `api/routes.py`, use `Depends(verify_firebase_token)` for protected endpoints.

4. **Adding a frontend tab**: Create a component in `frontend/src/features/results/components/`, import in `ResultsPage.tsx`.

5. **Chat capabilities**: The chatbot system prompt is self-contained in `routes.py::_SYSTEM_TEMPLATE`. Plot code runs sandboxed in a subprocess — never in the main process.
